# Prometheus v3.0 — Phases 1-11 Implementation Plan

**Status:** Phase 0 complete. This plan covers Phase 1 through Phase 11.

**Build Constraint (locked):** gcc/g++ is installed and on PATH. This enables `rusqlite` with `features = ["bundled"]` to compile SQLite from source without pre-built binaries. No `libsqlite3-dev` or vcpkg/MSVC static libs needed on Windows.

**Architecture Decision (locked):** Rust/Tauri owns the desktop OS layer (Kernel, PTY, HAL, window management, native APIs). React is presentation only. Python owns engineering logic, knowledge, simulation, and existing FastAPI surface. Rust Kernel dispatches desktop requests; Python sidecar (`prometheus --server`) handles platform APIs.

## Phase 1 — Workspace Completion (P1)

**Goal:** The desktop feels like a professional native application. VS Code-class polish.

### 1.1 Rust Kernel Service

New crate: `crates/prometheus-kernel`

```
Kernel
├── Terminal Manager (PTY)
├── Window Manager (desktop layout, not web window mgr)
├── Hardware Manager (USB/Serial/GPIO/etc.)
├── Filesystem Watcher
├── Event Bus (Rust side, bridges to Python SSE)
├── Plugin Runtime
└── Session Manager
```

**Task:** Create `crates/prometheus-kernel` with a `Kernel` struct that owns:
- `TerminalManager` — spawns PTY processes via `pty-process` or `portable-pty` crate
- `SessionManager` — persists workspace state (open windows, positions, terminal tabs) to SQLite via Tauri's `app.path`
- `EventBus` — in-process publish/subscribe that fans out to Tauri events for the webview

Tauri commands:
- `terminal_spawn(shell)` → returns session id
- `terminal_write(session_id, data)` → writes to PTY stdin
- `terminal_resize(session_id, cols, rows)` → resizes PTY
- `terminal_kill(session_id)` → kills PTY process
- `kernel_status()` → returns kernel health
- `session_save()` / `session_restore()` → workspace persistence

**Validation:** `cargo test -p prometheus-kernel` passes; Tauri dev build succeeds.

## Phase 2 — Hardware Platform (P2)

**Goal:** Prometheus talks to real engineering hardware. Rust HAL owns the transport layer.

### 2.1 Rust HAL Core

Current `crates/hal-core` is a stub with `SimulatedHal`. Expand:

**Transport traits (`crates/hal-core/src/transports/`):**
- `usb` — `usbd` crate, device enumeration, bulk/interrupt transfers
- `serial` — `serialport` crate, UART/RS232/RS485, custom framing protocol
- `gpio` — platform-specific sysfs/libudev (Linux), Win32 GPIO (Windows)
- `spi` / `i2c` — platform-specific
- `can` — `socketcan` (Linux), PCAN/USBCAN adapters
- `network` — TCP/UDP, already exists in Python layer
- `bluetooth` — `btleplug` / `hidapi`
- `jtag` / `swd` — OpenOCD libftdi/JLink bindings (C FFI)

**HAL registry:** `HalRegistry` maps `(transport, target_id)` → `Box<dyn HalDriver>`

**Rust Kernel integration:**
- `HardwareManager` in `prometheus-kernel` owns `HalRegistry`
- Tauri commands: `hardware_probe`, `hardware_connect`, `hardware_read`, `hardware_write`, `hardware_disconnect`
- Events: `hardware.connected`, `hardware.disconnected`, `hardware.data` streamed to frontend

### 2.2 Recovery Workflows (Python layer)

Recovery workflows stay in Python because they involve complex sequencing, database state, and existing engineering logic.

- Android: ADB, Fastboot, EDL, Odin mode → `hardware/recovery/android.py`
- Apple: DFU, Recovery mode → `hardware/recovery/apple.py`
- PC: BIOS, UEFI, TPM, Secure Boot → `hardware/recovery/pc.py`
- Embedded: JTAG/SWD flash, NAND/NOR/SPI → `hardware/recovery/embedded.py`

Python calls Rust HAL via existing REST API (`/epsilon/recovery/{device_id}`) or new Tauri commands if recovery needs direct hardware access.

### 2.3 Hardware Dashboard (Frontend)

Polish `HardwareApp.ts`:
- Real device tree view (not simulated telemetry)
- Transport status indicators (USB: connected/disconnected, BLE: scanning/advertising)
- Bus capture/log viewer: hex dump of raw transport frames
- Recovery workflow runner: button → streams progress via SSE

### Phase 2 Definition of Done

- `cargo test -p hal-core` passes with real transport tests (or hardware-in-the-loop if available)
- Device probe success >= 98% on test devices
- Recovery workflow success >= 95%
- Mean device diagnosis time <= 3 min
- Hardware dashboard shows live device tree

├── prometheus-kernel/     # New: OS layer (PTY, HAL, events, sessions)
│   └── Cargo.toml         # rusqlite with `bundled` feature (SQLite compiled from source via gcc)
├── aether-runtime/        # AI runtime (providers, routing, context, agents)

├── titan-core/            # Tokenizer, encoder/decoder (renamed from titan-tokenizer)

## Risks

1. **PTY on Windows:** Rust `pty-process` or `portable-pty` must support ConPTY (Windows 10+). Mitigation: test early in Phase 1.
2. **Rust/Python DB sharing:** Both need SQLite access. Mitigation: single writer (Python), Rust reads via same file, or SQLite WAL mode.
3. **Tauri event volume:** Streaming terminal output + hardware telemetry + SSE to frontend could overwhelm webview. Mitigation: batch events, throttle high-frequency sources.
4. **Phase 2 hardware diversity:** Real hardware is unavailable in CI. Mitigation: simulated HAL for tests, hardware-in-the-loop for manual validation.
